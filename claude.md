# Personal Website V3 — Migration Plan

## Context

Current state: A Create React App (client-side rendered) deployed to S3 with CloudFront. Blog content is fetched at runtime via Lambda + API Gateway from a private S3 bucket. The site returns an empty HTML shell ("You need to enable JavaScript to run this app") — all rendering happens client-side after the JS bundle downloads.

Target state: A Next.js site with static site generation (SSG), retro/terminal aesthetic inspired by stripe.dev, content authored in Notion and synced to the repo as markdown, deployed to S3 + CloudFront via automated GitHub Actions, with all AWS infrastructure defined as CDK (TypeScript).

Source repo: https://github.com/sesh11/personal-website-v2
Live site: https://nsesh.com

## Phase 1 — Next.js Application (TypeScript)

### Goal
Replace the CRA + Lambda architecture with a Next.js app that generates static HTML at build time.

### Key decisions
- Use Next.js App Router (not Pages Router) — it's the current standard
- Use TypeScript throughout
- Use Tailwind CSS for styling
- Blog posts are `.md` files in `/content` with YAML frontmatter
- Use `gray-matter` for frontmatter parsing, `remark` + `rehype` for markdown-to-HTML
- Use `next export` (static output mode) since we're deploying to S3, not Vercel

### What to port from v2
- Existing React components in `src/` — migrate to TSX, restructure for App Router
- Content from `CONTENT/` folder — convert to markdown with frontmatter if not already
- Any static assets in `public/`

### What to delete
- All Lambda function code
- API Gateway configuration/references
- Any runtime S3 fetch logic
- The private S3 content bucket dependency

### next.config.js key setting
```js
/** @type {import('next').NextConfig} */
const nextConfig = {
  output: 'export',        // Generates static HTML to /out
  images: {
    unoptimized: true,     // Required for static export (no image optimization server)
  },
}
module.exports = nextConfig
```

### Content parsing pattern (src/lib/content.ts)
```typescript
import fs from 'fs'
import path from 'path'
import matter from 'gray-matter'

const contentDir = path.join(process.cwd(), 'content')

export function getAllPosts() {
  const files = fs.readdirSync(contentDir).filter(f => f.endsWith('.md'))
  return files.map(filename => {
    const slug = filename.replace('.md', '')
    const raw = fs.readFileSync(path.join(contentDir, filename), 'utf-8')
    const { data, content } = matter(raw)
    return { slug, frontmatter: data, content }
  }).sort((a, b) => new Date(b.frontmatter.date).getTime() - new Date(a.frontmatter.date).getTime())
}

export function getPostBySlug(slug: string) {
  const raw = fs.readFileSync(path.join(contentDir, `${slug}.md`), 'utf-8')
  const { data, content } = matter(raw)
  return { slug, frontmatter: data, content }
}
```

### Blog post frontmatter format
```yaml
---
title: "Compound Engineering: A Methodology"
date: "2025-03-01"
summary: "How to use AI to reverse-engineer and modernize legacy systems"
tags: ["ai", "engineering", "methodology"]
---
```

### Validation
- `npm run build` produces a `/out` directory with fully rendered HTML
- Opening `/out/index.html` directly shows content (not a blank page)
- Each blog post has its own `/out/blog/[slug]/index.html`

## Phase 2 — Notion as Content Source

### Goal
Write exclusively in Notion and sync published content into the repo as markdown, triggering the existing build pipeline. Notion becomes the single authoring environment; the `content/` folder in git is a build artifact, not something you edit directly.

### How it works
1. Create a Notion database with properties: `Title` (title), `Slug` (text), `Date` (date), `Tags` (multi-select), `Status` (select: Draft / Published)
2. Write articles as Notion pages within this database
3. When ready to publish, set `Status` → "Published"
4. Run a sync script that pulls all published posts, converts Notion blocks to markdown with frontmatter, writes them to `content/`, and commits + pushes
5. The existing GitHub Actions pipeline handles the rest — build, deploy, invalidate cache

### Sync script (scripts/notion-sync.ts)
```typescript
// Dependencies: @notionhq/client, notion-to-md, gray-matter
// 1. Query Notion database: filter where Status == "Published"
// 2. For each page:
//    a. Read properties → build frontmatter (title, date, slug, tags)
//    b. Convert page blocks → markdown via notion-to-md
//    c. Combine frontmatter + markdown body
//    d. Write to content/{slug}.md
// 3. Git add, commit, push (or just write files and let the user push)
```

### Setup requirements
- Notion integration (internal) — create at https://www.notion.so/my-integrations
- Share your content database with the integration
- Store the Notion API key and database ID as environment variables:
  - `NOTION_API_KEY` — the integration token
  - `NOTION_DATABASE_ID` — the ID from the database URL

### npm dependencies
```bash
npm install @notionhq/client notion-to-md
```

### Usage
```bash
# One command to sync and deploy
npx tsx scripts/notion-sync.ts && git add content/ && git commit -m "sync from notion" && git push
```

### Repo additions
```
personal-website-v3/
├── scripts/
│   └── notion-sync.ts          # Notion → markdown sync script
├── .env.local                  # NOTION_API_KEY, NOTION_DATABASE_ID (git-ignored)
```

### Notes
- The `content/` folder is now generated output — avoid editing markdown files directly (Notion is the source of truth)
- Add a `.gitignore` comment or README note to make this clear
- Images in Notion pages: notion-to-md will extract Notion-hosted image URLs, but these expire. For durability, either download images to `public/images/` during sync, or host images separately (e.g., in your S3 bucket) and reference those URLs in Notion
- Can be wrapped as a Claude Code `/publish` command later for a single-command workflow

## Phase 3 — AWS Infrastructure as Code (CDK in TypeScript)

### Goal
Define all AWS hosting resources in TypeScript using CDK. One `cdk deploy` creates the entire stack.

### Resources to define
1. **S3 Bucket** — private, static website hosting (OAC access only)
2. **CloudFront Distribution** — CDN with OAC (Origin Access Control) to S3
3. **ACM Certificate** — HTTPS cert for nsesh.com (must be in us-east-1 for CloudFront)
4. **Route 53 Records** — A/AAAA alias records pointing nsesh.com to CloudFront
5. **IAM** — Deploy user/role for GitHub Actions

### CDK Stack outline (infra/lib/static-site-stack.ts)
```typescript
// Key resources:
// - s3.Bucket (private, no public access)
// - cloudfront.Distribution with S3 origin via OAC
// - certificatemanager.Certificate (DNS validated, us-east-1)
// - route53.ARecord aliased to CloudFront
// - Output the distribution ID and bucket name (needed by GitHub Actions)
```

### Setup commands
```bash
cd infra
npx cdk init app --language typescript
npm install @aws-cdk/aws-s3 @aws-cdk/aws-cloudfront @aws-cdk/aws-route53 @aws-cdk/aws-certificatemanager
```

### Notes
- ACM certificate MUST be in us-east-1 regardless of where the stack deploys
- Use OAC (not OAI) for S3 origin — OAI is legacy
- Export bucket name and CloudFront distribution ID as CloudFormation outputs so GitHub Actions can reference them

## Phase 4 — GitHub Actions CI/CD

### Goal
Fully automated deployments: push to main → site is live in ~60 seconds.

### Workflow 1: deploy-site.yml (triggers on push to main, excluding infra/)

```yaml
name: Deploy Site
on:
  push:
    branches: [main]
    paths-ignore:
      - 'infra/**'

permissions:
  id-token: write   # For OIDC
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm

      - run: npm ci
      - run: npm run build    # Generates /out

      - name: Configure AWS Credentials (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::ACCOUNT_ID:role/GitHubActionsDeployRole
          aws-region: us-east-1

      - name: Sync to S3
        run: aws s3 sync out/ s3://YOUR_BUCKET_NAME --delete

      - name: Invalidate CloudFront
        run: |
          aws cloudfront create-invalidation \
            --distribution-id YOUR_DIST_ID \
            --paths "/*"
```

### Workflow 2: deploy-infra.yml (triggers on infra/ changes)

```yaml
name: Deploy Infrastructure
on:
  push:
    branches: [main]
    paths:
      - 'infra/**'

permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install CDK dependencies
        run: cd infra && npm ci

      - name: Configure AWS Credentials (OIDC)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::ACCOUNT_ID:role/GitHubActionsDeployRole
          aws-region: us-east-1

      - name: CDK Deploy
        run: cd infra && npx cdk deploy --require-approval never
```

### Authentication approach: OIDC Federation (recommended over access keys)
- Create an IAM OIDC identity provider for GitHub Actions in your AWS account
- Create an IAM role (`GitHubActionsDeployRole`) that trusts the GitHub OIDC provider
- Scope the trust policy to your specific repo: `repo:sesh11/personal-website-v3:ref:refs/heads/main`
- No long-lived AWS credentials stored as GitHub secrets
- The role needs permissions: S3 (put/delete objects), CloudFront (create invalidation), and CDK bootstrap permissions for infra deploys

## Phase 5 — Design Overhaul (stripe.dev-inspired retro aesthetic)

### Goal
Terminal/retro aesthetic with modern performance. Think monospace typography, muted color palette, grid-based layout, ASCII-art-style decorations.

### Design elements from stripe.dev to study
- Monospace/terminal font (e.g., JetBrains Mono, IBM Plex Mono, Berkeley Mono)
- Dark background with high-contrast text
- ASCII figure references ("[ Fig. 1 ]", "10x")
- Table-like blog index with date, title, type columns
- Minimal color usage — mostly grayscale with one accent color
- Section markers using "/" prefix ("/Filter", "/Date", "/Topics")
- Subtle grid lines and borders

### Implementation
- Tailwind CSS for utility-first styling
- Custom CSS for terminal-specific effects (scanlines, cursor blink, etc.)
- Keep it CSS-only where possible — no heavy animation libraries
- Responsive: the retro look should degrade gracefully on mobile
